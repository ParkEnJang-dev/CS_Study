



- 쿠키는 클라이언트에 저장된는 작은 데이터 입니다.
- 세션은 서버에 사용자 상태를 저장하고, 그식별자만 클라이언트에 전달하는 방식입니다.

### Cookie
---
- 쿠키는 서버가 클라이언트에 저장하도록 지시하는 작은 데이터 조각입니다.이후 요청마다 자동으로 서버에 전송합니다.

- 동작흐름
	1. 클라이언트가 서버에 요청
	2. 서버가 응답 헤더에 Set-Cookie 포함
	3. 브라우저가 쿠키를 로컬에 저장
	4. 같은 도메인 요청 시 쿠키를 자동 전장

- 저장 되는 정보
	- 사용자 식별자 
	- 로그인 여부 플래그
	- 설정 값 (언어, 테마 등)
	- 민감 정보(비밀번호, 개인정보는) 저장하지 않은
	- 보안 관련 속성
		- HttpOnly: JS에서 접근 불가 → XSS 방어
		- Secure: HTTPS에서만 전송
		- SameSite: CSRF 공격 완화
		- Expires / Max-Age: 쿠키 수명 제어

### Session
---

- 서버가 사용자별 상태를 저장하는 공간
- 클라이언트는 세션 ID만 들고 있고, 실제 데이터는 서버에 있습니다.
- 요청 시 세션 ID로 서버 상태 조회
### JWT(Json Web Token)
---
- JWT는 브라우저에 종속된 쿠키 기반 세션 방식의 한계를 보완하기 위해 등장한 토큰 기반 인증 방식이다. JSON 형식의 토큰을 사용하므로 다양한 전송 프로토콜과 환경에서 활용 가능하며, 중앙 인증 서버와의 조합을 통해 SSO 및 MSA 환경에 적합하다. 기본적으로 서버의 세션 상태를 요구하지 않는 인증 구조를 제공하지만, 보안 및 운영 요구에 따라 추가적인 상태 관리가 필요할 수 있다.
  
- JWT에서 Access Token / Refresh Token을 분리하는 이유
- **역할 분리**
	- Access Token: API 요청 인증용
	- Refresh Token: Access Token 재발급 전용

- **보안 강화**
	- Access Token을 짧은 수명으로 운용해 탈취 피해 최소화
		- Refresh Token 노출 빈도를 최소화하여 핵심 자격 증명 보호
- **사용자 경험 유지**
	- Access Token 만료 시 재로그인 없이 자동 재발급 가능

- **운영 통제력 확보**
	- Refresh Token을 상태 관리하여 강제 로그아웃, 권한 변경 대응 가능

- **무상태성과 제어의 균형**
	- Access Token은 stateless 검증
		- 인증 판단한때 서버 저장 상태가 개입되지 않는다.
	- Refresh Token은 stateful 관리
		- 서버가 토큰 상태를 저장하고 추적 한다.
